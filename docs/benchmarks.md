# Benchmarks

AMP publishes measurable performance baselines for its hot paths. These are
regenerated by `cargo bench -p amp-benches` and enforced by the `perf-regression`
CI job (Phase 2.7).

The latency SLO is **p99 < 1 ms matchmaking @ 10 000 active matches / 1 000
concurrent connections** (server-internal, excluding network).

## Matchstore microbenchmark (`amp-benches/benches/matchstore.rs`)

Proves the Phase 2.1 root-cause fix for the latency regression. The active-match
store was `Arc<ArcSwap<HashMap<String, ActiveMatch>>>`, whose every mutation
cloned the ENTIRE map inside `rcu`. At `MAX_ACTIVE_MATCHES = 10_000` that was a
~1.5â€“2 MB allocation + memcpy per insert/update/archive â€” structurally
incompatible with a sub-millisecond SLO.

### Baseline (commit `prodpath` Phase 2; commodity Linux dev box)

| Operation @ 10k entries | Old (`ArcSwap` full clone) | New (`DashMap`) | Speedup |
|:---|---:|---:|---:|
| `insert` | 1.44 ms | 364 ns | ~3,950Ă— |
| `update` | 1.46 ms | 72 ns | ~20,300Ă— |
| `remove` | ~1.4 ms | 238 ns | ~6,000Ă— |

A single settlement under the old design did `update` + `archive` + `insert`
mutations, i.e. **~4 ms of map cloning** for one match â€” exceeding the full p99
budget on its own. The `DashMap` path makes the matchstore contribution to a
matchmaking tick well under 1 Âµs total.

### Reproducing

```bash
cargo bench -p amp-benches --bench matchstore
```

## End-to-end load test (`amp-loadtest`)

Full pipeline â€” connect â†’ EIP-191 challenge login â†’ requestMatch â†’
submitOutcome (real EIP-712 signature) â†’ on-chain settlement â€” against a local

### Baseline (200 concurrent clients, 1 worker, loopback, 8 s run)

| Stage | p50 | p99 | max |
|:---|---:|---:|---:|
| connect | 7.4 ms | 18.6 ms | 18.9 ms |
| login (challenge + ECDSA) | 44.2 ms | 67.0 ms | 67.2 ms |
| **match (requestMatch RPC)** | **28.5 ms** | **52.4 ms** | **53.1 ms** |
| total (connectâ†’settle) | 98.6 ms | 109.0 ms | 109.4 ms |

Result: **200 logins â†’ 192 matches found â†’ 96 successful settlements** (1
outcome per completed pair; losers' duplicate submissions are correctly
rejected).

**What dominates match p99:** the matchmaker tick cadence (50 ms poll,
`amp-server/src/main.rs`) + loopback RPC round-trips â€” **not** the matchstore,
which contributes ~72 ns per operation (see microbenchmark above). Before
Phase 2.1 the matchstore alone took ~1.4 ms per mutation, exceeding the entire
budget on its own. The tick is a tunable matchmaking cadence, not processing
latency.
