# Roadmap

Our mission is to become the default settlement layer for all competitive games on Avalanche.

> The bounded, phase-by-phase path to mainnet lives in [`prodpath.md`](../prodpath.md).
> This page is the higher-level milestone view.

## 2026 Milestones

### Q1: The Foundation (Completed)
- [x] Core Rust `amp-server` (Matchmaker & Verifier) with Cap'n Proto RPC.
- [x] Solidity `AMPRegistry` + `AMPSettlement` contracts with OpenZeppelin
  `Ownable2Step`, `Pausable`, `ReentrancyGuard`, and `EIP712`.
- [x] Match lifecycle: create, join, cancel, expire, dispute resolution.
- [x] Challenge-response authentication (`requestChallenge` + `login`).
- [x] `IndexedQueue` matchmaker with MMR-sorted buckets, binary search, 50ms tick.
- [x] Native SDKs: C++, C#, Go, Python, Rust (JS via napi-rs added in Q2).
- [x] `amp-relayer` with sled-backed queue, EIP-1559 gas, nonce tracking, retry + dead-letter.
- [x] Docker + docker-compose for all services.
- [x] CI with cargo fmt, clippy, test, and Forge test.

## Resolved limitations (previously listed as open)
- **Player dedup in match queue:** `IndexedQueue::push` replaces a player's stale
  entry — a player can no longer be matched into two matches.
- **Glicko-2 NaN/Inf / iteration runaway:** the Go solver has `MaxGlickoIterations`
  and NaN/Inf guards; the Rust solver is hardened identically.
- **Ghost matches:** the matchmaker skips pairings whose partner dropped while queued,
  and disconnect notification fans out to live opponents.
- **Persistence schema versioning:** sled values carry `SCHEMA_VERSION`; the server
  refuses to boot on mismatch (opt-in wipe available). ( sled→redb migration in Phase 3. )

## Long-term Vision (2027+)
Beyond 1v1 and small arena games, AMP aims to support massive-scale competitive play, including Battle Royales and persistent world territorial control, all settled trustlessly on Avalanche.
